## 2. 目的・スコープ

### 2.1 目的

人間がブラウザで行う操作（遷移・入力・クリック・スクロール・画面からの情報取得）を **Playwright for Python** で再現し、定型処理の自動化やデータ取得パイプラインの入力源とする。

### 2.2 スコープ（本仕様で扱う範囲）

- **Playwright** の `Browser` / `BrowserContext` / `Page` のライフサイクル（起動・終了・Cookie・ストレージ）
- **Locator** および自動待機を前提としたページ操作と待機戦略（**固定 `sleep` 依存を避ける**。§3.5 ②）
- **Storage State** 等によるログインセッションの再利用（§3.5 ③）
- **ヘッドレス／ヘッドフル**の切り替え（§3.5 ①）
- 取得データの正規化とログ出力
- **SQLite への永続化**（スキーマ定義・マイグレーション・バックアップ方針を含む）
- **常設 PC 上での継続運用**（スケジュール実行、電源・スリープ・リモート保守の前提を含む）
- **スクレイピング／自動化検知の緩和**（レート制御、待機のジッタ、ブロック兆候時の停止・冷却）
- **Stealth（任意）** — 初期化スクリプト等による自動化シグナルの緩和（§3.6、要件 ST）
- **対象サイトがログインを要するジョブ**に対する、アカウントの用意・運用の前提（§4.3、要件 AC）
- 失敗時のリトライ・レポート

### 3.1 Playwright for Python

ブラウザ自動化は **Playwright for Python** に統一し、API・ディレクトリ構成・設定キーは **Playwright に最適化**する。

| 項目 | 内容 |
|------|------|
| パッケージ | `playwright`（ブラウザは `playwright install` で取得。**本番ジョブの既定エンジンは Chromium** とし、`playwright install chromium` を前提にする） |
| API 形態 | **同期 API**（`sync_playwright`）を既定とする。高スループットが必要な場合のみ **非同期 API**（`async_playwright` + `asyncio`）を採用し、その場合は SQLite 同時アクセスも `aiosqlite` 等で設計する |
| 中核概念 | **Browser** → **BrowserContext**（セッション単位・Cookie・権限）→ **Page**（タブ相当）。永続ログインは **`launch_persistent_context`** または **`storage_state` の保存／読込**で行う |
| 要素操作 | **Locator**（`page.get_by_role` / `get_by_text` / `locator` 等）を優先し、**自動待機**と **Assertions（expect）** を活用する。安易な固定 `sleep` や生のセレクタ文字列の乱用は避ける |
| 待機・タイムアウト | `Page.set_default_timeout` / `set_default_navigation_timeout` を一括設定し、ジョブ内で単位を揃える |
| デバッグ・証跡 | スクリーンショット（`page.screenshot`）、**Trace**（`tracing.start` / `stop` → zip）、任意で **HAR** や **動画**（`record_video_dir`） |
| 設定ファイル | Node 用の `playwright.config.ts` に相当する記述は **Python 側の `config/`（YAML 等）** に集約し、起動オプションへマッピングする |

### 3.2 永続化：SQLite

| 項目 | 内容 |
|------|------|
| 採用 | **SQLite 3**（標準ライブラリ `sqlite3`、または SQLAlchemy 等のドライバ経由） |
| ファイル | 例: `data/app.db`。パスは設定または環境変数で指定し、リポジトリには含めない（`.gitignore`） |
| 接続 | アプリ起動時にパスを解決。複数プロセスから同一ファイルへの同時書き込みは避ける（単一ワーカー前提）。**常設 PC 1 台に集約する構成と SQLite の相性が良い** |
| 推奨 PRAGMA | 起動時に `journal_mode=WAL` を検討（読み取りと書き込みの競合を緩和）。`foreign_keys=ON` で参照整合性を有効化 |
| スキーマ変更 | **Alembic**（SQLAlchemy 利用時）またはバージョン付き SQL スクリプトでマイグレーションを管理する |
| バックアップ | 定期コピーまたは `sqlite3` の `.backup`、運用頻度は別紙 |

### 3.3 補助ライブラリ（必要に応じて）

- **DB アクセス**: 標準 `sqlite3`、または **SQLAlchemy 2.x** + **Alembic**（スキーマ管理を重視する場合）
- **環境変数・設定**: `pydantic-settings` など
- **非同期**: Playwright の `async` API を利用する場合は `asyncio` を前提に設計（非同期 SQLite は `aiosqlite` 等を別途検討）
- **ログ**: 標準 `logging` またはプロジェクト標準のロガー

### 3.4 スクレイピング／自動化検知への緩和（設計方針）

Web 側ではアクセス頻度・パターン・セッション挙動などから **ボットらしさ** を検知する場合があります。本仕様では **相手サーバーへの負荷を抑え、機械的すぎる規則性を減らす** ことで、誤検知・レート制限を減らすことを目標とする（**完全な回避を保証するものではない**）。

| 方針 | 内容 |
|------|------|
| レート制御 | ページ遷移・クリック等の **最小間隔** と、単位時間あたりの **最大アクション数** を設定で上限化する |
| ジッタ | 待機時間に **乱数幅（ジッタ）** を付与し、同一ミリ秒間隔の連続アクセスを避ける |
| セッション一貫性 | ログインや Cookie を使う場合は **`launch_persistent_context(user_data_dir=...)`** または **`BrowserContext.storage_state` の JSON 保存／読込**でセッションを再利用し、不要な再ログインを減らす |
| 並列の抑制 | 同一サイト・同一オリジンに対し、**複数ブラウザを同時に立てて負荷を掛けない**（キューで直列化する等） |
| ブロック兆候 | HTTP **429**、ログイン強制、チャレンジ画面などを **ブロック兆候** とみなし、即座に連続アクセスを止め **冷却時間** を置く |
| 挙動 | スクロールは一括ではなく **段階的** に送る等、人の操作に近い手順を取る（実装はジョブごとに定義） |
| ヘッドレス | 環境によってはヘッドレスが検知されやすい場合がある。**検証時は `headless=False`** で挙動を比較できるようにする（BR-01 と併用） |

### 3.5 ヘッドレス／ヘッドフル、待機戦略、Storage State

本節は実装・運用時の判断に使う。**詳細な API は Playwright 公式ドキュメント**に従う。

#### ① ヘッドレスとヘッドフル（Headless / Headful）

| モード | 説明 | 想定用途 |
|--------|------|----------|
| **ヘッドレス**（`headless=True`） | ブラウザウィンドウを **表示せず** バックグラウンドで実行する。**GPU 負荷が小さく、常設 PC・サーバー向き** | 本番ジョブ、スケジュール実行、無人運用 |
| **ヘッドフル**（`headless=False`） | **実際にブラウザウィンドウを表示**して実行する | **デバッグ**、挙動の目視確認、サイト側がヘッドレスを検知しやすい場合の**比較検証**（人間が操作しているように見せる必要がある運用の検討時を含む） |

- 既定は **ヘッドレス**とし、必要に応じ設定でヘッドフルに切り替える（要件 BR-01）。

#### ② 待機（Wait）戦略

固定秒数の **`time.sleep(3)` のような待ち**だけに頼らない。ページ負荷やネットワークで表示タイミングが変わるため、**「特定の要素が操作可能になるまで待つ」**制御を基本とする。

| 方針 | 内容 |
|------|------|
| 推奨 | **Locator** の操作（`click` / `fill` 等）は **自動待機**されるため、まず Locator と **`expect`**（`playwright.sync_api.expect`）を使う |
| セレクタ待ち | 従来型の **`page.wait_for_selector(".class_name")`** のように、**特定セレクタの要素が現れるまで待ってから**次の処理へ進めることができる。表示済みを確認してから `click` すると意図が明確になる |
| 避けるべき例 | 理由なく **`page.wait_for_timeout` を連打**する、`time.sleep` だけで画面同期を代用する（遅すぎる／短すぎる両方のリスク） |

- タイムアウトの既定値は **`Page.set_default_timeout`** で一元管理する（要件 BR-02、OP-02）。

#### ③ 認証情報の保持（Storage State）

**毎回ログイン処理を走らせる**と、対象サービス側で **アカウント制限・凍結のリスクが高まりやすい**場合がある。一度ログインできたあとの **Cookie およびローカルストレージ等の状態**をファイルに保存し、次回以降は **ログイン済み状態で `BrowserContext` を起動**できるようにする。

| 手段 | 内容 |
|------|------|
| **`storage_state`** | `BrowserContext.storage_state(path="...")` で JSON を保存し、次回 `browser.new_context(storage_state="...")` で読み込む。**クレデンシャルそのものではなくセッション状態**を保存する |
| **永続プロファイル** | `launch_persistent_context(user_data_dir=...)` でユーザーデータディレクトリを固定し、ブラウザプロファイル単位で状態を保持する |
| 運用上の注意 | `storage_state` ファイルには **セッション情報が含まれる**ため、**`.gitignore`** とファイル権限で漏えいを防ぐ。パスは設定 `storage_state_path` / `user_data_dir` で管理する（要件 BR-04、AD-04） |

### 3.6 Stealth（自動化検知緩和の補助）

**Stealth** とは、ページ上の JavaScript が参照する **自動化らしさのシグナル**（例: `navigator.webdriver` 等）を抑えたり、挙動を通常のブラウザ利用に近づけたりする **補助的手段**の総称である。本仕様では **Playwright 公式機能の外**に置き、**§3.4 のレート制御・§3.5 のモード／待機／Storage State と併用**しても、**検知やブロックを防げる保証はない**。

| 項目 | 内容 |
|------|------|
| 実装の典型 | 各 **`Page`** または **`BrowserContext`** に対し、**`add_init_script`** で Stealth 用の JavaScript を注入する。またはコミュニティ提供の **Stealth 対応ラッパ／スニペット**を `playwright_factory` で一括適用する |
| 採用方針 | **既定は無効**（`stealth_enabled=false`）。有効化する場合は **設定で明示**し、ジョブログに **Stealth オン／オフ**を残す |
| メンテナンス | Playwright・Chromium の **バージョンアップで挙動が変わりやすい**。採用するスクリプト名・版を **`requirements.txt` と文書で固定**し、更新時は **回帰試験**を必須とする |
| セキュリティ | 外部配布のスクリプトを取り込む場合は **改ざん防止**（チェックサム・社内ミラー）と **出所の記録**を行う |
| 関係 | **§3.4（検知緩和）** の補助。**AD 要件**（レート・冷却・Storage State）を **代替しない** |

## 4. システム構成

### 4.1 運用形態：常設 PC をサーバーとして利用

本システムは **常時稼働を前提とした PC（以下「常設 PC」）1 台**をサーバー代わりとし、ブラウザ自動化ジョブ・SQLite・ログ・アーティファクトを **同一マシン上**に置く運用とする。クラウド VM や複数ノードへの分割は本仕様の既定としない。

| 項目 | 内容 |
|------|------|
| 役割 | スケジュールまたは常駐プロセスによるジョブ実行、データ蓄積、必要に応じたリモートからの保守・結果取得 |
| 配置 | アプリ・仮想環境・DB ファイル・`artifacts/` は常設 PC のローカルディスク上（ネットワークドライブ上の DB は避ける） |
| ネットワーク | インターネット接続はジョブ要件に従う。リモート操作は **RDP（Windows）・SSH（Linux）** 等、社内ポリシーに沿った手段を定義する |
| OS 更新 | Windows Update 等による **意図しない再起動**でジョブが中断されないよう、更新タイミングと再起動ポリシーを運用で決める |
| 電源 | **スリープ・休止状態を無効化**し、夜間ジョブが止まらないようにする（省電力要件がある場合は別途検討） |
| 可用性 | 停電対策として **UPS** の利用を検討する。ハード故障時は SQLite ファイルのバックアップからの復旧手順を別紙で定義する |
| アクセス制御 | 常設 PC は業務用アカウント・ディスク暗号化（BitLocker 等）・不要な共有の無効化など、**物理盗難・不正ログイン**に備える |

### 4.3 対象サイトが認証を要する場合のアカウント運用（前提）

ログインセッションを **`storage_state`** 等で再利用するジョブでは、**技術だけではなく「どのようなアカウントを用いるか」**が停止・制限の頻度に影響する。以下は **運用上の一般論**であり、**対象サービスの利用規約・法令・社内ポリシーに従う**こと。特定プラットフォームへの適合を保証しない。

| 観点 | 内容 |
|------|------|
| **複数アカウント** | 単一アカウントへの負荷集中による **制限・凍結リスク**を下げるため、**ローテーション可能な複数の業務用アカウント**を用意する運用が一般的である（いわゆる **使い捨て／バーナー的なサブアカウント**を複数持つ運用を含む）。SQLite 側では **どの `storage_state`／`user_data_dir` がどのアカウントに対応するか**を追跡できるようにする（要件 AC） |
| **電話／メール等の認証** | **未認証のまま**の新規アカウントは、自動アクセス開始後 **短時間で利用制限を受けやすい**との知見がある。**SMS・メール等で認証済み**のアカウントを用いる |
| **作成からの経過時間** | **作成直後のみ**のアカウントより、**数ヶ月程度「寝かせた」**（通常利用の履歴を伴う）アカウントの方が、停止されにくい場合がある |
| **プロフィールの最低限の整備** | **アイコン未設定・投稿ゼロのみ**など極端に空のプロフィールは、自動化・スパム検知で **優先的に制限されやすい**との知見がある。**適切なアイコンと少数の公開投稿**等、**通常利用に近い外観**を備えておく |
| **Storage State との関係** | 上記は **§3.5 ③・BR-04・AD-04** で定めるセッション保存と組み合わせる。**認証済み・安定したアカウント**で一度ログインし、その **`storage_state` を再利用**する方が、繰り返しログインを挟むより負荷が小さい場合が多い |

## 5. 機能要件

### 5.1 Playwright：Browser / Context / セッション

| ID | 要件 |
|----|------|
| BR-01 | `chromium.launch(headless=...)` または `chromium.launch_persistent_context(...)` で、設定により **ヘッドレス／ヘッドフル**を切り替え可能とする（§3.5 ①） |
| BR-02 | `Page.set_default_timeout` および `Page.set_default_navigation_timeout` を設定から注入し、**ナビゲーション・Locator 操作の待機上限**を一元管理する |
| BR-03 | `Browser.new_context` または `launch_persistent_context` の引数で **viewport**・**locale**・**timezone_id**・**user_agent**・**extra_http_headers** 等を必要に応じて指定可能とする（対象サイトのポリシーに従う） |
| BR-04 | ログイン状態の再利用は **`storage_state` ファイルの読み書き**、または **`user_data_dir` 付き persistent context** のいずれか（または併用方針）を文書化し、保存パス・権限を定義する（§3.5 ③）。**毎回ログイン**を避け、不要な認証試行によるアカウント負荷を抑える |
| BR-05 | ジョブ終了時は **`BrowserContext.close`**（および **`browser.close`**）を確実に呼び、リソースリークを防ぐ |

### 5.2 Playwright：Page / Locator と抽出

| ID | 要件 |
|----|------|
| OP-01 | `Page.goto`、**Locator** によるクリック・入力・`press`、フォーム送信、**`mouse.wheel` 等によるスクロール**を用いる |
| OP-02 | 待機は **`Locator` の自動待機**および **`expect`（`playwright.sync_api.expect`）** を優先する。必要に応じ **`page.wait_for_selector(...)`** で要素の出現を明示的に待ってから操作する。**`time.sleep` や `page.wait_for_timeout` の常時利用**に依存しない（§3.5 ②） |
| OP-03 | 取得結果は辞書またはデータクラスにマッピングし、スキーマをコードまたは型で明示する |
| OP-04 | セレクタ文字列は **`selectors.py` 等に集約**し、可能な範囲で **ロール・テキストベースの Locator**（`get_by_role`、`get_by_text`）を優先して DOM 変更に強くする |
| OP-05 | スクリーンショットは **`Page.screenshot`**（フルページオプション含む）を標準とし、失敗時調査（RL-02）に使う |

### 5.3 信頼性

| ID | 要件 |
|----|------|
| RL-01 | 一時的な失敗に対し、指数バックオフ等のリトライ方針を定義する（上限回数あり） |
| RL-02 | 失敗時は URL・**`Page.screenshot`**・スタックトレース（社内ポリシー内）を残せるようにする。必要に応じ **Trace zip**（`tracing`）を併用する |
| RL-03 | 同一ジョブの冪等性：再実行しても二重登録が起きないよう、抽出キーと DB 制約を検討する |

### 5.4 可観測性

| ID | 要件 |
|----|------|
| OB-01 | ジョブ開始・終了・主要ステップを構造化ログで出力する |
| OB-02 | 必要に応じ **`BrowserContext.tracing`（start / stop）** で Trace（`.zip`）を `artifacts/` に保存可能とする。再現性が必要なジョブでは採用を検討する |

### 5.5 SQLite 永続化

| ID | 要件 |
|----|------|
| DB-01 | データベースファイルのパスを設定で指定可能とし、既定パスを文書化する |
| DB-02 | 業務キー（例: 外部ソース ID）に **UNIQUE 制約**を張り、再実行時の二重登録を防ぐ |
| DB-03 | ジョブ実行単位で `run_id` 等を記録し、いつ・どのバッチで書き込んだか追跡できる |
| DB-04 | スキーマ変更はマイグレーションで行い、手作業の直接 ALTER に依存しない |
| DB-05 | 本番相当データを扱うファイルはバックアップ方針に従う（コピー時は WAL 整合性に注意） |

### 5.6 常設 PC サーバー運用

| ID | 要件 |
|----|------|
| SV-01 | ジョブは **常設 PC 上**で実行する。別 PC から同一 SQLite を直接共有して同時書き込みしない |
| SV-02 | OS のスリープがジョブ実行中に発生しないよう、電源設定を文書化し運用する |
| SV-03 | 定期ジョブを用いる場合は **タスクスケジューラ（Windows）／cron・systemd timer（Linux）** 等の起動方法を定義する |
| SV-04 | リモートからの操作・ログ取得手順（接続方法・権限あるアカウント）を文書化する |

### 5.7 検知緩和・トラフィック制御

| ID | 要件 |
|----|------|
| AD-01 | 主要な操作（遷移・クリック・送信等）の **直前に待機** を入れ、**最小間隔** と **ジッタ（最大幅）** を設定で指定可能とする |
| AD-02 | **1 時間あたり**（および必要なら **1 日あたり**）の **最大ページビュー数または最大アクション数** を設定で上限化し、超過時はジョブを一時停止または翌枠へ延期する |
| AD-03 | **ブロック兆候**（例: HTTP 429、繰り返し失敗するナビゲーション、ログイン強制・チャレンジ画面の表示）を検知した場合、同一セッションでの無制限リトライを行わず **冷却タイマー** を発動し、ログに理由を残す |
| AD-04 | Cookie／ローカルストレージを使う場合は **`launch_persistent_context(user_data_dir=...)`** または **`context.storage_state(path=...)` の読み書き**でセッションを維持し、ジョブ間で不必要にログインし直さない |
| AD-05 | 同一オリジンに対する **同時実行ブラウザ数** を 1 に制限する、またはセマフォで直列化する（メモリ制約と合わせて定義） |
| AD-06 | AD-01〜AD-03 のパラメータは **環境別**（開発／本番）で値を分け、本番は保守的な既定値から開始できること |
| AD-07 | 冷却中・レート超過によるスキップは **可観測性ログ**（OB-01）に明示的に記録する |

### 5.8 Stealth（任意）

| ID | 要件 |
|----|------|
| ST-01 | **`stealth_enabled`** が真のときのみ、Context／Page 生成時に Stealth 用の **`add_init_script`**（または採用ライブラリが定める同等のフック）を適用する |
| ST-02 | Stealth を **無効**にした場合と **有効**にした場合で、同一ジョブの挙動比較ができるよう、ログにフラグを出力する |
| ST-03 | Stealth 用スクリプトは **`src/browser/stealth_init.js`** 等の **リポジトリ管理ファイル**、または **版固定した依存パッケージ**のいずれかに限定し、未レビューのコピペを本番に入れない |
| ST-04 | Playwright または Chromium を **マイナーバージョンアップ**した際は、Stealth 有効ジョブを **回帰試験**対象に含める |
| ST-05 | Stealth は **§3.4・要件 AD の代替にならない**（レート・冷却・Storage State を省略しない） |

### 5.9 認証ジョブ用アカウント（追跡・運用）

| ID | 要件 |
|----|------|
| AC-01 | ログインを要するジョブでは、**利用するアカウント識別子**（例: 社内管理 ID、メールのハッシュ等）と **`storage_state_path`／`user_data_dir`** の対応を **文書または DB** で追跡できるようにする |
| AC-02 | **複数アカウント**を用いる場合は、**ローテーション順序・切替条件**（制限検知時のスキップ等）を定義し、ログに残す |
| AC-03 | 新規取得アカウントを本番ジョブに投入する前に、**§4.3** に沿った状態（認証済み、経過時間、プロフィール）であることを **チェックリスト**等で確認できること |
| AC-04 | アカウント資格情報（パスワード等）は **コードに埋め込まず**、常設 PC 上の **権限付き保管**（シークレットストア・暗号化ストレージ等）に従う |
| AC-05 | §4.3 の運用は **対象サービスの規約に適合する範囲**で行うこと。本仕様は **違反を助長するものではない** |

## 6. 非機能要件

| 区分 | 内容 |
|------|------|
| 実行環境 | **本番は常設 PC 1 台**上の OS（Linux / Windows のいずれかを明記）。開発用 PC は別でもよいが、本番パス・権限に合わせて検証する |
| 常設 PC | 24h 稼働想定。熱対策・埃・振動に配慮した設置。長期無人運用時は **遠隔からの死活監視**（最低限：ディスク残量・ジョブ成否ログ）を検討する |
| 同時実行 | 同一マシンでの **Playwright の `Browser` 同時起動数**および **Context 数**の上限（メモリ・CPU に依存）。原則 **1 ジョブ 1 Browser（または 1 永続 Context）** を推奨 |
| Playwright | **Locator・自動待機・expect** のベストプラクティスに従い、API のバージョンは `playwright` とブラウザバイナリをセットで固定する |
| SQLite | **単一ライター**が基本。複数プロセスから同一 DB ファイルへの同時書き込みは設計上避ける。**常設 PC 単体構成**によりネットワーク越しの DB 競合を避ける。読み取りは WAL 下で並行しやすい |
| スループット | 1 時間あたりの操作回数の上限（対象サイトへの負荷・レート制限を考慮） |
| 検知緩和 | **単調な間隔・過剰な並列・短時間の大量アクセス**を避ける（§3.4、要件 AD）。対策でも **検知・ブロックをゼロにできる保証はない** |
| Stealth | **任意**。有効時は **§3.6・要件 ST** に従い、公式非サポートであること・**アップグレードで破損しうる**ことを前提にメンテナンスコストを見込む |
| セキュリティ | 認証情報はコードに埋め込まず、環境変数またはシークレット管理に委ねる。DB ファイルのファイル権限・保管場所を適切に設定する |
| 保守 | Python・Playwright・ブラウザ・（利用する場合は SQLAlchemy/Alembic）のバージョンを `requirements.txt` 等で固定 |

## 9. エラーハンドリング方針

| 分類 | 例 | 扱い |
|------|-----|------|
| 一時的 | タイムアウト、一時的なネットワークエラー | リトライ後、失敗ならログして終了または次ジョブへ |
| 恒久的 | **Locator がタイムアウト**（DOM 変更・セレクタ不整合） | リトライしない。アラート・修正タスク化 |
| ブロック兆候 | HTTP 429、チャレンジ／ログイン強制の繰り返し等 | **無制限リトライしない**。冷却（AD-03）後、設定見直しまたは手動確認 |
| ビジネス | 対象データなし | 正常系として空結果を返す |

## 10. テスト方針

- **単体テスト**: Locator 文字列以外の純粋関数・正規化ロジックをモックで検証
- **結合テスト**: ステージング URL または固定 HTML を `page.goto("file://...")` やローカル HTTP サーバで再現（可能な範囲）
- **E2E**: CI では **ヘッドレス Chromium** でスモーク。必要に応じ **`pytest-playwright`** を検討する。本番相当の頻度は別パイプラインで調整
